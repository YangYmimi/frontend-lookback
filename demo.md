### HTTP

> 应用层协议，默认端口 `80`. 无连接无状态.

* HTTP请求分为三部分：状态行、请求头、消息主体；

* HTTP响应分为三部分：状态行、响应头、响应正文；

* `请求-应答` 模式：即每个请求/应答客户与服务器之间都要新建一个连接，完成之后立即断开；

* `Keep-Alive` 模式：客户与服务器之间连接持久有效（但是不是一直有效，存在 timeout，max等设置），保持了当前 TCP 的连接。使得 `管线化机制（多个 HTTP 请求整批提交的技术）` 成为可能。

```
Keep-Alive: timeout=5, max=100
=>
timeout=5表示可以持续5s；max=100表示最多接收100个请求之后就断开连接
```

```
请求1 -> 响应1 -> 请求2 -> 响应2 -> 请求3 -> 响应3
=> 管线化后
请求1 -> 请求2 -> 请求3 -> 响应1 -> 响应2 -> 响应3
```

#### 优点

* 连接可以复用

* 支持响应分块

* 缓存控制机制

#### 缺点

* 通信使用明文（不加密），内容容易被篡改

* 不验证通信方的身份，因此可能遭遇伪装

* 无法证明通信报文的完整性，所以可能被篡改

#### HTTP Headers

* `Content-Type`: 指示资源的MIME类型。服务端根据请求头中的 `Content-Type` 字段得知请求中的消息主体是用何种方式编码，从而可以对消息主题进行解析。

* `Transfer-Encoding`: 规定传输所采用的编码形式。如果一个HTTP消息（请求或者应答）的 `Transfer-Encoding=chunked`，则说明消息体由数量未定的块组成，并以一个大小为 0 的块结束。

  - 优势：可以边生成内容边传输内容

  - 缺点：无法得知下载进度，只能通过 chunked 大小为 0 来得知传输结束。
